# KossJS 沙箱实现计划

> **供智能体工作者参考：** 必需的子技能：使用 superpowers:subagent-driven-development（推荐）或 superpowers:executing-plans 来逐任务实现本计划。步骤采用复选框（`- [ ]`）语法以便追踪。

**目标：** 将《安全与沙箱设计.md》v1.3 落地为 KossJS 的细粒度能力掩码、同步/异步审核回调、JS 审核 API、取消、调试和测试覆盖。

**架构：** 保持现有 `KossInstance` 单线程拥有 Boa `Context` 的模型，不引入跨线程直接访问 `Context`。新增 `sandbox` 模块集中管理能力、审核状态、参数序列化、错误类型和 C ABI 数据结构，`runtime.rs` 只负责在 JS API 和 C ABI 边界调用沙箱决策函数。先实现同步审核闭环，再实现异步审核和取消，最后逐步把所有受保护 API 接入统一检查点。

**技术栈：** Rust 2024、Boa `0.21.1`、Tokio、C ABI、Python `ctypes`、pytest、Cargo test。

---

## 当前代码基线

- `src/runtime.rs` 目前只有 5 个粗粒度 capability：`KOSS_CAP_FS`、`KOSS_CAP_NET`、`KOSS_CAP_CRYPTO`、`KOSS_CAP_WORKER`、`KOSS_CAP_EXTERNAL_LOADER`。
- `include/kossjs.h` 与当前 5 位 capability 对齐，但尚未包含 v1.3 文档中的 28 个细粒度 capability、审核回调 API、取消 API、调试 API。
- `KossInstance` 当前字段为 `context`、`event_loop`、`worker_pool`、`external_module_loader`、`capabilities`，尚无 audit 状态字段。
- `register_native_bindings()` 和 `koss_get_binding()` 只按模块名做粗粒度开关，`handle_binding()` 返回的每个 binding 方法没有逐方法 capability 控制。
- `register_internal_module_loader()` 已检查 `KOSS_CAP_EXTERNAL_LOADER`，但没有对应 `MODULE_LOAD` 细粒度位和审核回调。
- `register_native_fetch()` 只受 `KOSS_CAP_NET` 的注册阶段控制，调用阶段没有 `NET_FETCH` 检查和审核。
- `register_senri_ffi_impl()` 无 capability 开关，`_senri_ffi.open`、`func`、`alloc` 等高风险入口需要接入 `FFI_*` 位。
- `kossjs_interface.py` 已支持 `capabilities` 参数，但常量仍是当前 5 位模型；没有 `check_sandbox`、`check_sandbox_async`、`cancel_audit`、`enable_audit_debug` 封装。
- `kossjs_interface.ts` 只封装 `eval/version/destroy`，后续可低优先级同步新增常量和基础审核 API。

## 文件结构与职责

- Create: `src/sandbox.rs`
  定义细粒度 capability 常量、组合常量、`SandboxState`、审核回调类型、异步配置、取消 token、参数序列化、统一决策函数、错误构造辅助函数。
- Modify: `src/lib.rs`
  导出 `sandbox` 模块。
- Modify: `src/runtime.rs`
  将 capability 常量迁移到 `sandbox`，在 `KossInstance` 增加 `sandbox: SandboxState`，在 C ABI 和 JS 原生函数入口调用 `sandbox` 检查。
- Modify: `include/kossjs.h`
  同步 C ABI：细粒度 capability、审核回调 typedef、配置 struct、注册/取消/调试 API。
- Modify: `kossjs_interface.py`
  同步常量、ctypes 原型、Python 侧同步审核封装、异步审核封装、取消、调试开关。
- Modify: `kossjs_interface.ts`
  同步 capability 常量和最小 C ABI wrapper；TS 不作为第一阶段验收阻塞项。
- Modify: `src/bindings.rs`
  只在需要时调整 binding 方法名称或参数提取。优先避免大规模改动此文件，把检查放在 `runtime.rs` 暴露给 JS 的边界。
- Modify: `src/_senri_ffi/mod.rs`
  给 `_senri_ffi.open`、`func`、`funcAsync`、`closeAsync`、内存/结构体相关入口接入 `FFI_*` 和审核。
- Modify: `src/_senri_ffi/memory.rs`
  给 alloc/free/address/read/write 等内存相关函数接入 `FFI_ALLOC`。
- Create: `test/test_sandbox_capabilities.py`
  覆盖细粒度 capability 的注册阶段和调用阶段行为。
- Create: `test/test_sandbox_audit.py`
  覆盖同步审核、JS 审核、拒绝错误类型、参数转换、pwd 语义的基础行为。
- Create: `test/test_sandbox_async_audit.py`
  覆盖异步审核、超时、并发、取消、重入限制。
- Modify: `test/conftest.py`
  如有必要，增加创建指定 capability 实例的 fixture。
- Modify: `README.md`
  更新 capability 示例和安全说明。
- Modify: `KossJS_Docs/zh/security-sandbox/security-sandbox.md`
  同步用户文档。

    match target {

        "fs.mkdir" | "fs.mkdirSync" => Some(FS_MKDIR),
        "fs.rename" | "fs.renameSync" => Some(FS_RENAME),
        "fs.chmod" | "fs.chmodSync" | "fs.chown" | "fs.chownSync" | "fs.symlink" | "fs.link" => Some(FS_CHMOD),

        "net.connect" | "net.createConnection" => Some(NET_TCP_CLIENT),
        "net.createServer" => Some(NET_TCP_SERVER),
        "crypto.createHash" => Some(CRYPTO_HASH),
        "crypto.createHmac" => Some(CRYPTO_HMAC),
        "crypto.randomBytes" | "crypto.randomUUID" | "crypto.getRandomValues" => Some(CRYPTO_RANDOM),
        "crypto.pbkdf2" | "crypto.pbkdf2Sync" => Some(CRYPTO_PBKDF2),

pub unsafe extern "C" fn koss_check_sandbox(
    ptr: *mut KossInstance,
    callback: crate::sandbox::AuditCallback,
    userdata: *mut c_void,

    output_license_once();

        instance.sandbox.sync_userdata = userdata;

    }
}

- [ ] **Step 2: 实现深度限制和循环处理**

在 `src/sandbox.rs` 或 `runtime.rs` 中实现参数转换规则：

- `null` -> `"null"`
- `undefined` -> `"undefined"`
- 函数 -> `"[Function]"`
- 对象/数组 -> JSON 序列化，最大深度 10，循环引用为 `"[Circular]"`
- Buffer -> 默认完整字符串；脱敏模式下大 Buffer 为 `"<Buffer size=N>"`
- 其他 -> `String(value)`

建议复用 JS 侧 JSON replacer，避免直接遍历 Boa 对象导致复杂度过高。生成临时 JS 函数：

```javascript
(function(value, redact) {
  var seen = new WeakSet();
  function clean(v, depth) {
    if (v === null) return null;
    if (typeof v === 'undefined') return 'undefined';
    if (typeof v === 'function') return '[Function]';
    if (typeof Buffer !== 'undefined' && Buffer.isBuffer && Buffer.isBuffer(v)) {
      return redact && v.length > 256 ? '<Buffer size=' + v.length + '>' : v.toString('utf8');
    }
    if (typeof v !== 'object') return v;
    if (seen.has(v)) return '[Circular]';
    if (depth >= 10) return '[MaxDepth]';
    seen.add(v);
    if (Array.isArray(v)) return v.map(function(x) { return clean(x, depth + 1); });
    var out = {};
    Object.keys(v).forEach(function(k) { out[k] = clean(v[k], depth + 1); });
    return out;
  }
  return JSON.stringify(clean(value, 0));
})
```

## 已知实现取舍

- `koss_eval` 是宿主主动执行 JS 的入口，不受 `DYNAMIC_CODE` 限制；`DYNAMIC_CODE` 限制 JS 内部 `eval`、`Function` 和字符串定时器。
- 异步审核第一版可以阻塞等待 C callback completion，以保证语义正确；真正 Promise 挂起模型可以作为后续性能优化。
- 旧 5 位 capability 建议保留为兼容别名，但新文档、测试、示例只使用 v1.3 细粒度常量。
- Worker 在 v1.3 文档中没有单独 capability 位；第一版可保留旧 `KOSS_CAP_WORKER` 兼容行为，后续应在设计文档中明确是否新增 `WORKER` 位或归入 `DYNAMIC_CODE`。

## 自查结果
