# 安全与沙箱设计（v1.4）

本文档定义 KossJS 沙箱系统的完整设计，包含 **能力位掩码**、**审核掩码** 与 **审核回调** 三层安全机制。

---

## 一、能力位掩码

### 1.1 设计原则

KossJS 默认行为与 Node.js 一致：**信任运行在其中的 JS 代码**，赋予其完整的文件系统、网络和加密能力。对于运行 **不可信 JS 代码** 的场景（如第三方插件引擎、用户脚本沙箱），必须通过能力位掩码限制每个实例的权限。

能力位掩码是 **静态权限声明**，在实例创建时确定，运行时不可更改。所有危险 API 均受能力位控制，未授予的能力在 JS 层表现为 `undefined` 或调用时抛出 `TypeError`。

**能力位容量说明**：当前使用 32 位无符号整数（`uint32_t`），已占用 28 位。若未来能力位超过 32 个，将升级为 `uint64_t`，并保持向后兼容（通过检测高位掩码）。

### 1.2 能力位定义

```c
// 文件系统（6 个细粒度操作）
#define FS_READ         (1 << 0)   // 读取文件、目录、元数据
#define FS_WRITE        (1 << 1)   // 写入、追加、截断、复制
#define FS_DELETE       (1 << 2)   // 删除文件或空目录
#define FS_MKDIR        (1 << 3)   // 创建目录
#define FS_RENAME       (1 << 4)   // 重命名/移动
#define FS_CHMOD        (1 << 5)   // 修改权限、所有者、创建链接

// 网络（5 个细粒度操作）
#define NET_TCP_CLIENT  (1 << 6)   // TCP 客户端连接与数据发送
#define NET_TCP_SERVER  (1 << 7)   // TCP 服务端监听
#define NET_UDP         (1 << 8)   // UDP 发送与接收
#define NET_DNS         (1 << 9)   // DNS 解析
#define NET_FETCH       (1 << 10)  // HTTP/HTTPS 客户端请求

// 加密（4 个细粒度操作）
#define CRYPTO_HASH     (1 << 11)  // 哈希（MD5, SHA1, SHA256 等）
#define CRYPTO_HMAC     (1 << 12)  // HMAC
#define CRYPTO_RANDOM   (1 << 13)  // 随机数生成（randomBytes, randomUUID）
#define CRYPTO_PBKDF2   (1 << 14)  // 密钥派生（pbkdf2）

// 内置 FFI（5 个细粒度操作）
#define FFI_OPEN        (1 << 15)  // 加载/关闭动态库
#define FFI_CALL        (1 << 16)  // 调用 C 函数（同步/异步）
#define FFI_ALLOC       (1 << 17)  // 内存分配/释放/地址获取
#define FFI_CALLBACK    (1 << 18)  // 创建 JS 回调指针
#define FFI_STRUCT      (1 << 19)  // 结构体、数组、指针操作

// 其他模块（8 个操作）
#define NATIVE_ADDON    (1 << 20)  // 加载 .node 原生模块
#define WASM            (1 << 21)  // WebAssembly
#define SHARED_MEMORY   (1 << 22)  // SharedArrayBuffer 与 Atomics
#define HIGHRES_TIME    (1 << 23)  // 高精度计时（performance.now, process.hrtime）
#define SYSINFO         (1 << 24)  // 系统信息泄露（os.hostname, os.cpus 等）
#define MODULE_LOAD     (1 << 25)  // JS 模块加载（require, import）
#define DYNAMIC_CODE    (1 << 26)  // 动态代码执行（eval, Function, setTimeout 字符串）
#define DEBUG           (1 << 27)  // 调试/内省（Error.stack, console 详细输出）

// 组合常量
#define KOSS_CAP_SANDBOX 0
#define KOSS_CAP_ALL_FS  (FS_READ | FS_WRITE | FS_DELETE | FS_MKDIR | FS_RENAME | FS_CHMOD)
#define KOSS_CAP_ALL_NET (NET_TCP_CLIENT | NET_TCP_SERVER | NET_UDP | NET_DNS | NET_FETCH)
#define KOSS_CAP_ALL_CRYPTO (CRYPTO_HASH | CRYPTO_HMAC | CRYPTO_RANDOM | CRYPTO_PBKDF2)
#define KOSS_CAP_ALL_FFI (FFI_OPEN | FFI_CALL | FFI_ALLOC | FFI_CALLBACK | FFI_STRUCT)
#define KOSS_CAP_ALL    0xFFFFFFFF   // 所有能力位（当前定义均在此范围内）
```

### 1.3 能力位检查规则

- **模块注册阶段**：根据能力位决定是否将全局 API 注册到 JS 环境。例如，若 `NET_FETCH` 未设置，则 `fetch` 函数为 `undefined`。
- **API 调用阶段**：每个原生实现内部必须检查对应的能力位。若未授权，应抛出 `KossCapabilityError` 或静默失败（返回 `undefined`）。
- **能力位与审核回调的关系**：
  - 若能力位禁止该操作 → 直接拒绝（JS 层 API 不存在，或调用时立即失败）
  - 若能力位允许 → 进入审核回调链路

### 1.4 预设配置示例

```c
// 完全沙箱（仅纯计算）
KossInstance *inst = koss_create_with_caps(KOSS_CAP_SANDBOX);

// 仅允许读取文件（不允许写入、删除等）
uint32_t caps = FS_READ;
KossInstance *inst = koss_create_with_caps(caps);

// 允许 TCP 客户端和 HTTP 请求，但不允许 TCP 服务器、UDP、DNS
uint32_t caps = NET_TCP_CLIENT | NET_FETCH;
KossInstance *inst = koss_create_with_caps(caps);

// 允许所有加密操作，但禁用 PBKDF2（防御计算密集型攻击）
uint32_t caps = KOSS_CAP_ALL_CRYPTO & ~CRYPTO_PBKDF2;
```

## 二、审核掩码

审核掩码是 **动态审核策略声明**，在实例创建时确定，运行时可更改。它与能力位掩码使用相同的位定义，但语义不同：

- **能力位掩码**：控制"是否允许"（静态权限）
- **审核掩码**：控制"是否需要审核"（动态策略）

### 2.1 审核掩码定义

审核掩码使用与能力位掩码相同的 28 个位定义（见 1.2 节），但语义为：

- **位设置为 1**：该 API 需要经过审核回调
- **位设置为 0**：该 API 不需要审核（直接放行，前提是能力位允许）

**默认审核掩码**：`0`（不审核任何 API）。宿主必须显式设置审核掩码，才能启用审核回调。

> 审核掩码只能设置在能力位掩码已授予的位上。若审核掩码包含了能力位掩码未授予的位，这些位将被视为无效（忽略，不会报错），因为对应的 API 已被禁用，无需审核。

### 2.2 审核掩码与能力位掩码的关系

```
如果能力位禁止 → 直接拒绝（KossCapabilityError），不触发审核
如果能力位允许 + 审核位设置 → 触发审核回调
如果能力位允许 + 审核位未设置 → 直接放行
```

**重要约束**：
- 审核的 API 必须是已启用的（能力位允许的）
- 默认情况下，禁用的 API 会覆盖审核（即禁用优先于审核）
- 宿主可以随时更改审核掩码，但不能更改能力位掩码（实例创建时确定）

### 2.3 审核掩码 API

```c
/**
 * 设置审核掩码
 * 
 * @param ptr   KossInstance 指针
 * @param mask  审核掩码（使用与能力位相同的位定义）
 * @return KossResult
 * 
 * 示例：
 *   // 只审核文件系统读取和网络请求
 *   koss_set_audit_mask(inst, FS_READ | NET_FETCH);
 *   
 *   // 审核所有文件系统操作
 *   koss_set_audit_mask(inst, KOSS_CAP_ALL_FS);
 *   
 *   // 不审核任何 API（直接放行）
 *   koss_set_audit_mask(inst, 0);
 */
KOSS_API KossResult koss_set_audit_mask(KossInstance* ptr, uint32_t mask);

/**
 * 获取当前审核掩码
 * 
 * @param ptr   KossInstance 指针
 * @return 当前审核掩码
 */
KOSS_API uint32_t koss_get_audit_mask(KossInstance* ptr);
```

### 2.4 预设审核配置示例

```c
// 完全审核（所有已启用的 API 都需要审核）
koss_set_audit_mask(inst, KOSS_CAP_ALL);

// 只审核高风险 API（FFI、动态代码、原生模块）
koss_set_audit_mask(inst, FFI_OPEN | FFI_CALL | FFI_ALLOC | FFI_CALLBACK | FFI_STRUCT | DYNAMIC_CODE | NATIVE_ADDON);

// 只审核文件系统操作
koss_set_audit_mask(inst, KOSS_CAP_ALL_FS);

// 不审核任何 API（直接放行）
koss_set_audit_mask(inst, 0);
```

### 2.5 JS 侧审核掩码 API


## 三、审核回调机制

能力位掩码提供 **粗粒度静态权限**，审核掩码提供 **细粒度审核策略**，而审核回调提供 **动态审核决策**，允许宿主在运行时根据上下文（函数名、参数、文件路径）决定是否放行某个 API 调用。

### 3.1 审核回调的生命周期

1. **注册时机**：审核回调可以随时注册（但通常在实例创建后、执行代码前注册）
2. **生效范围**：注册后对所有需要审核的 API（根据审核掩码）生效
3. **可变性**：可以随时取消或替换审核回调
4. **实例隔离**：每个实例有独立的审核回调，互不影响

> 当 KossInstance 被 `koss_destroy` 销毁时，所有关联的审核回调将被自动释放，无需手动清除。

### 3.2 审核回调模式

KossJS 支持两种审核回调模式：

- **同步审核**：回调同步返回 `bool`，适合快速决策（如路径检查）。
- **异步审核**：回调返回 `Promise<bool>` 或通过 C 异步回调，允许执行 I/O、用户确认等异步操作。KossJS 会挂起原始 API 调用，直到 Promise 解决或异步回调完成。

### 3.3 整体决策链路（同步与异步）

决策流程：

```
JS 调用受保护 API（例如 fs.readFile）
    │
    ▼
┌─────────────────────────────────┐
│ 能力位掩码检查                    │
└─────────────────────────────────┘
    │
    ├── 未设置 → 直接拒绝（KossCapabilityError）
    │
    ▼ 已设置
┌─────────────────────────────────┐
│ 审核掩码检查                      │
└─────────────────────────────────┘
    │
    ├── 未设置 → 直接放行
    │
    ▼ 已设置
┌─────────────────────────────────┐
│ 是否存在外部审核回调？            │
└─────────────────────────────────┘
    │
    ├── 无 → 进入 JS 审核
    │
    ▼ 有
┌─────────────────────────────────┐
│ 根据注册类型调用外部审核回调       │
│ （同步或异步）                    │
└─────────────────────────────────┘
    │
    ├── 同步回调返回 false / 异常 → 拒绝
    ├── 同步回调返回 true → 进入 JS 审核
    ├── 异步回调 → 等待完成
    │      ├── 超时/取消/返回 false → 拒绝
    │      └── 返回 true → 进入 JS 审核
    │
    ▼
┌─────────────────────────────────┐
│ 是否存在 JS 审核回调？            │
└─────────────────────────────────┘
    │
    ├── 无 → 放行
    │
    ▼ 有
┌─────────────────────────────────┐
│ 调用 JS 审核回调                 │
│ （同步或异步，自动识别）           │
└─────────────────────────────────┘
    │
    ├── 同步返回 false / 异常 → 拒绝
    ├── 同步返回 true → 放行
    ├── 异步（Promise）→ 等待解决
    │      ├── 解决 false / 拒绝 → 拒绝
    │      └── 解决 true → 放行
    │
    ▼
    放行，执行实际 API 调用
```

**要点：**
- 能力位是 **第一道闸门**，未授权时审核回调不被触发。
- 审核掩码是 **第二道闸门**，未设置时直接放行。
- 外部回调与 JS 回调是 **串联** 关系：外部必须放行，JS 才可能执行。
- 多次调用注册函数会 **覆盖** 之前的回调（每个实例最多一个外部回调和一个 JS 回调）。
- **JS 审核回调也支持异步**（返回 Promise），以允许脚本执行异步检查（例如访问网络 API 验证权限）。

### 3.4 性能分级建议

并非所有 API 都需要经过审核回调。对于高频、低风险的 API，建议仅通过能力位控制，避免回调开销。

| API 类别 | 推荐控制方式 | 理由 |
|----------|-------------|------|
| `console.log`, `Date.now`, `Math.random` | 仅能力位（或默认允许） | 调用频次极高，审核开销不可接受 |
| `fs.readFile`, `net.connect` | 能力位 + 同步审核 | 频次中等，通常需要路径/域名检查 |
| `_senri_ffi.open`, `process.dlopen` | 能力位 + 异步审核（可选） | 高风险，可能需要用户确认或复杂验证 |
| 宿主注册的自定义函数 | 由宿主自行决定 | 宿主完全控制 |

设计文档不强制规定，但建议实现者遵循此原则。

### 3.5 C 侧 API（同步与异步，含控制参数）

#### 3.5.1 同步审核回调

```c
/**
 * 注册同步外部审核回调
 * 
 * @param ptr   KossInstance 指针
 * @param func  回调函数指针。传入 NULL 表示清除回调。
 * @param userdata  用户数据
 * @return KossResult
 * 
 * 回调函数原型：
 *   typedef bool (*AuditCallback)(const char* target, const char** args, int argc, const char* pwd, void* userdata);
 * 
 * 参数解释：
 *   - target   : API 名称（见 2.5 命名规范）
 *   - args     : 字符串数组（参数已序列化）
 *   - argc     : 参数个数
 *   - pwd      : 当前模块目录（绝对路径），未知则为 NULL
 *   - userdata : 注册时传入的用户数据
 * 
 * 返回值：true = 允许，false = 拒绝
 * 异常：回调抛出 C++ 异常 → 视为 false，并记录日志（若调试模式开启）
 */
KOSS_API KossResult koss_check_sandbox(KossInstance* ptr, AuditCallback func, void* userdata);
```

#### 3.5.2 异步审核回调（增强版）

```c
/**
 * 异步审核回调完成时的通知函数类型
 * 
 * @param result  审核结果（true = 允许，false = 拒绝）
 * @param userdata  用户数据（与注册时的 userdata 相同）
 */
typedef void (*AuditCompletion)(bool result, void* userdata);

/**
 * 异步审核回调函数原型
 * 
 * 宿主实现此函数，在异步操作完成后调用 on_complete。
 * 如果宿主超时或内部错误，应调用 on_complete(false)。
 * 
 * @param target  同同步回调
 * @param args    同同步回调
 * @param argc    同同步回调
 * @param pwd     同同步回调
 * @param userdata  用户数据（注册时传入）
 * @param on_complete  完成通知函数
 * @param on_complete_userdata  传递给完成通知的用户数据
 */
typedef void (*AuditCallbackAsync)(
    const char* target,
    const char** args,
    int argc,
    const char* pwd,
    void* userdata,
    AuditCompletion on_complete,
    void* on_complete_userdata
);

/**
 * 异步审核配置参数
 */
typedef struct {
    uint64_t timeout_ms;        // 审核超时时间（毫秒），0 表示无超时（默认）
    uint32_t max_concurrent;    // 最大并发异步审核数，0 表示使用默认值 64
    bool allow_reentrant;       // 是否允许审核回调中再次触发审核（默认 false）
    bool queue_when_full;       // 并发满时是否排队（true）或直接拒绝（false），默认 true
    uint32_t max_reentrant_depth; // 最大重入深度（仅当 allow_reentrant=true 时有效），0 表示使用默认值 5
} AuditAsyncConfig;

/**
 * 注册异步外部审核回调（增强版）
 * 
 * 调用此函数后，所有受保护 API 将采用异步审核模式。
 * 如果同时注册同步和异步回调，异步优先。
 * 
 * @param ptr   KossInstance 指针
 * @param func  异步回调函数指针。传入 NULL 表示清除。
 * @param config 配置参数（可为 NULL，使用默认值）
 * @param userdata  用户数据
 * @return KossResult
 */
KOSS_API KossResult koss_check_sandbox_async_ex(
    KossInstance* ptr,
    AuditCallbackAsync func,
    const AuditAsyncConfig* config,
    void* userdata
);
```

**默认配置**：
- `timeout_ms = 0`（无超时）
- `max_concurrent = 64`（经验值，适用于大多数场景；宿主可根据硬件能力调整，如低端设备设为 16）
- `allow_reentrant = false`
- `queue_when_full = true`
- `max_reentrant_depth = 5`（仅当 `allow_reentrant = true` 时生效）

**异步审核实现要点：**
- KossJS 会为每个待审核的 API 调用分配一个挂起点，等待 `on_complete` 被调用。
- **超时控制**：如果 `timeout_ms > 0` 且宿主未在超时内调用 `on_complete`，KossJS 自动视作拒绝（`false`），并释放挂起状态。超时发生时，若调试模式开启，会记录警告。
- **并发限制**：`max_concurrent` 指定允许同时进行的最多异步审核数。超出限制的新审核请求将被排队（如果 `queue_when_full = true`）或立即拒绝（若 `false`）。防止资源耗尽。
- **重入控制**：默认不允许审核回调中再次触发审核（`allow_reentrant = false`）。若设置为 `true`，则允许有限重入，深度受 `max_reentrant_depth` 限制。
- **取消支持**：提供取消 API（见 3.6 节）。

### 3.6 异步审核取消 API

本节描述取消 API，对应 3.5.2 中提到的取消支持。

```c
/**
 * 取消指定 target 的待处理异步审核（所有匹配的）
 * 
 * @param ptr    KossInstance 指针
 * @param target 要取消的 API 名称（NULL 表示取消所有待处理审核）
 * @return 取消的审核数量
 */
KOSS_API uint64_t koss_cancel_audit(KossInstance* ptr, const char* target);

/**
 * 取消所有待处理的异步审核
 */
static inline uint64_t koss_cancel_all_audits(KossInstance* ptr) {
    return koss_cancel_audit(ptr, NULL);
}

/**
 * 为特定审核请求分配一个 cancel token，以便精确取消
 * 
 * 在注册异步审核时可获取 token，宿主可在任何时机调用取消函数。
 */
typedef void* AuditCancelToken;

/**
 * 扩展的异步审核配置（含 token 输出）
 */
typedef struct {
    uint64_t timeout_ms;
    uint32_t max_concurrent;
    bool allow_reentrant;
    bool queue_when_full;
    uint32_t max_reentrant_depth;
    AuditCancelToken* out_token;   // 输出参数，用于接收 token
} AuditAsyncConfigEx;

/**
 * 使用扩展配置注册异步审核，并通过 out_token 获取取消令牌
 */
KOSS_API KossResult koss_check_sandbox_async_ex2(
    KossInstance* ptr,
    AuditCallbackAsync func,
    const AuditAsyncConfigEx* config,
    void* userdata
);

/**
 * 通过令牌取消特定的待审核请求
 * 
 * @param token 之前获得的令牌
 * @return true 表示成功取消（请求尚在等待中），false 表示已完成或不存在
 */
KOSS_API bool koss_cancel_audit_by_token(AuditCancelToken token);
```

**取消语义**：
- 取消一个尚未完成的审核请求后，对应的 API 调用将被拒绝（抛出 `KossSecurityError`），且审核回调的 `on_complete` **不会再被调用**。
- 已完成的审核不可取消。
- 取消操作是线程安全的（异步审核可能在不同线程完成通知）。

### 3.8 target 命名规范与参数转换规则

#### target 命名规范

| API 类型 | target 格式 | 示例 |
|---------|-------------|------|
| Node.js 内置模块方法 | `模块名.方法名` | `fs.readFile`, `crypto.createHash` |
| FFI 操作 | `_senri_ffi.操作名` | `_senri_ffi.open`, `_senri_ffi.alloc` |
| 动态库函数调用 | `<库名>.<函数名>` | `libc.system` |
| 全局函数 | `函数名` | `fetch`, `setTimeout` |
| 宿主注册的函数 | 注册时使用的名称 | `db.query`, `sendEmail` |

#### 参数转换规则

- `null` → `"null"`
- `undefined` → `"undefined"`
- 函数 → `"[Function]"`
- 对象/数组 → JSON 序列化（限制深度 10，循环引用替换为 `"[Circular]"`）
- 大 Buffer → `"<Buffer size=N>"`
- 其他基本类型 → `String(value)`

### 3.9 参数隐私与安全警告

> **警告**：审核回调会接收到 API 参数的字符串表示。如果参数中包含敏感信息（如密码、Token、私钥），这些信息会以明文形式传递给回调。若回调实现存在漏洞（例如被恶意 JS 覆盖），可能导致敏感数据泄露。
>
> **建议**：
> - 宿主不应将高敏感数据直接作为参数传递给受保护 API；应通过其他安全通道传递。
> - 宿主可在回调中实现参数过滤，例如对某些 API 的参数进行脱敏（截断、哈希等）。
> - KossJS 提供可选的“参数脱敏模式”：通过 `koss_enable_argument_redaction(ptr, true)` 启用后，对于 `Buffer` 和字符串长度超过 256 的参数，仅显示长度或前缀，不传递完整内容。**此模式默认关闭**。

### 3.10 重入检测与有限重入实现细节

KossJS 为每个实例维护一个**重入计数器**（基于线程局部存储或调用栈深度）。规则：

- 初始值为 0。
- 每次进入审核回调（外部或 JS）时，计数器 +1；退出时 -1。
- 如果 `allow_reentrant == false` 或计数器已达 `max_reentrant_depth`（仅当 `allow_reentrant == true` 时有效）：
  - 任何新的受保护 API 调用**不会触发审核**，而是直接拒绝（抛出 `KossSecurityError`），并在调试模式下记录警告。
- 如果 `allow_reentrant == true` 且计数器 < `max_reentrant_depth`：
  - 允许继续触发审核。
  - 递归审核时，每个嵌套层级都会独立调用审核回调，宿主需要自己处理避免死循环。

**重入深度配置**：`max_reentrant_depth` 可在 `AuditAsyncConfig` 中设置（默认 5），既防止无限递归，也为特殊场景（如需要多次嵌套审核）提供灵活性。

**重入场景示例**：
```javascript
koss.checkSandbox((target, args, pwd) => {
    if (target === 'fs.readFile') {
        // 这里调用 fs.statSync 会触发新的审核
        const stat = require('fs').statSync(args[0]);
        return stat.isFile();
    }
    return true;
});
```
- 如果 `allow_reentrant = false`，上述 `statSync` 调用将直接拒绝（抛出错误）。
- 如果 `allow_reentrant = true`，则会递归审核 `statSync`，但受限于 `max_reentrant_depth`。

**实现建议**：
- 在 `KossInstance` 结构中增加 `audit_reentrant_counter` 字段（原子整数）。
- 在审核回调入口检查该计数器和配置，决定是否允许进入。
- 在 `on_complete` 或同步返回时递减计数器。

### 3.11 调试模式

为便于排查审核回调的错误，KossJS 提供调试模式开关：

```c
/**
 * 启用或禁用审核调试模式
 * 
 * @param enable  true = 开启，false = 关闭
 * 
 * 开启后：
 *   - 同步/异步回调中抛出的异常会输出到 stderr（或宿主日志回调）
 *   - 审核被拒绝的原因会附带额外错误信息
 *   - 异步审核的超时或挂起会记录警告
 *   - 重入拒绝会记录当前深度和配置的最大深度
 * 
 * 生产环境应关闭调试模式，避免信息泄露。
 */
KOSS_API void koss_enable_audit_debug(KossInstance* ptr, bool enable);
```

JS 侧也可通过环境变量或 `process.env.KOSS_AUDIT_DEBUG` 控制，但推荐使用 C API。

### 3.12 关于 `pwd` 参数语义

- `pwd` 表示 **触发本次 API 调用的 JS 模块所在的目录（绝对路径）**。
- 对于通过 `koss_run_file` 执行的入口文件，`pwd` 为该文件所在目录。
- 对于 `require()` 或 `import()` 加载的模块，`pwd` 为**当前模块文件所在目录**（与 `__dirname` 一致）。
- 如果调用来自 `eval`、`new Function` 或 `koss_run_string`，`pwd` 为 `NULL`。
- 异步回调同样遵守此语义。

实现要求：KossJS 必须在每个模块加载时记录其路径，并在调用受保护 API 前设置当前上下文 `pwd`。

### 3.13 与能力位掩码的联合使用（典型配置）

| 场景 | 能力位掩码 | 审核掩码 | 审核回调 | 效果 |
|------|-----------|---------|---------|------|
| 完全禁止文件系统 | `FS_READ=0` | 任意 | 无 | 直接报错 |
| 允许文件系统，限制路径 | `FS_READ=1`, `FS_WRITE=1` | `FS_READ` | 检查路径 | 可读可写，但只允许安全目录 |
| 允许网络，限制域名 | `NET_FETCH=1` | `NET_FETCH` | 检查域名 | 只允许 fetch 白名单域名 |
| 允许 FFI，需用户确认 | `FFI_OPEN=1` | `FFI_OPEN` | 异步弹窗 | 加载动态库需交互确认 |

## 四、错误类型

审核拒绝时抛出的异常类型：

- `KossCapabilityError`：因能力位掩码禁止而拒绝。
- `KossSecurityError`：因审核回调拒绝（外部或 JS 回调返回 `false` 或 Promise 解决为 `false`）。
- `KossTimeoutError`：异步审核超时导致拒绝。（当异步审核超过配置的 `timeout_ms` 时抛出。）
- `KossCancelError`：异步审核被取消导致拒绝。（当审核被 `cancelAudit` 或令牌取消时抛出。）

JS 代码可据此区分拒绝原因。

### 4.1 错误消息优化建议

**生产环境**：使用通用错误消息，避免信息泄露。
```javascript
// 生产环境
throw new KossSecurityError("Access denied");
```

**调试模式**：使用详细错误消息，便于排查问题。
```javascript
// 调试模式
throw new KossSecurityError("KossSecurityError: sandbox audit denied for fs.readFile (path: /etc/passwd)");
```

**建议实现**：
- 默认使用通用错误消息
- 通过 `koss_enable_audit_debug(true)` 启用详细错误消息
- 错误消息中不包含敏感信息（如密码、Token）

## 五、完整使用示例（Python 绑定）

```python
from kossjs import KossJS

# 同步外部审核回调
def sync_audit(target: str, args: list[str], pwd: str | None) -> bool:
    if target == "fs.readFile":
        return args[0].startswith("/tmp/sandbox/")
    return True

# 异步外部审核回调（模拟用户确认）
def async_audit(target: str, args: list[str], pwd: str | None, done):
    # 假设这里弹出对话框，最终调用 done(True/False)
    allowed = show_confirm_dialog(f"Allow {target} with args {args}?")
    done(allowed)

# 配置异步审核（含重入深度、并发、超时）
config = {
    "timeout_ms": 30000,          # 30秒超时
    "max_concurrent": 10,         # 最多10个并发审核
    "allow_reentrant": True,      # 允许有限重入
    "queue_when_full": True,
    "max_reentrant_depth": 3      # 最大重入深度3
}

# 创建实例：允许文件系统读取 + 网络 fetch
caps = KossJS.FS_READ | KossJS.NET_FETCH
koss = KossJS(capabilities=caps)

# 设置审核掩码：只审核文件系统读取操作
koss.set_audit_mask(KossJS.FS_READ)

# 注册异步审核回调（优先于同步）
koss.check_sandbox_async(async_audit, config)

# 开启调试模式
koss.enable_audit_debug(True)

# 执行 JS 代码，其中包含异步审核
koss.run_string("""
    koss.checkSandbox(async (target, args, pwd) => {
        // 二次审核：仅允许 .txt 文件
        if (target === 'fs.readFile') return args[0].endsWith('.txt');
        return true;
    });
    
    // 创建取消令牌（可选）
    const token = koss.createAuditCancelToken();
    setTimeout(() => token.cancel(), 5000);  // 5秒后取消
    
    const data = require('fs').readFileSync('/tmp/sandbox/secret.txt');
    console.log(data.toString());
""")

# 主动取消所有待审核
koss.cancelAudit()
```
