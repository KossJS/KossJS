## 1. 设计总览与架构分层

### 1.1 核心设计原则

1. **命名空间隔离**：所有运行时提供的 API 必须通过 `koss:` 协议显式导入，不产生全局污染（除了极少数必要的全局对象）。
2. **分层解耦**：兼容层（Node/Bun/Deno）互相独立，均可单独启用，互不依赖。
3. **特性驱动加载**：通过独立的 `Builtin Flags` 控制模块是否可被解析，未启用的模块导入直接报错。
4. **零底层侵入**：所有兼容层完全在 JS 层实现，仅调用已有的 `__koss_*` 内部绑定。
5. **沙箱独立**：`Builtin Flags` 仅控制 API 是否可见，实际权限由 `Capability Bits` 独立控制。

### 1.2 三层架构模型

```
┌─────────────────────────────────────────────────────────────────┐
│ L3: 用户态兼容层 (Userland Shims)                               │
│ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐            │
│ │ koss:node/*  │ │   koss:bun   │ │  koss:deno   │            │
│ │ (Node风格)   │ │  (Bun风格)   │ │  (Deno风格)  │            │
│ └──────┬───────┘ └──────┬───────┘ └──────┬───────┘            │
│        │                 │                 │                    │
│        └────────┬────────┴────────┬────────┘                    │
│                 ▼                 ▼                             │
│ ┌──────────────────────────────────────────────────────────────┐│
│ │ L2: 内部系统层 (Internal System Layer)                       ││
│ │ koss:internal/*  - 基础绑定封装，无风格，纯函数式            ││
│ │ 导出: readFileSync, writeFileSync, httpServe, tcpConnect等  ││
│ └──────────────────────────┬───────────────────────────────────┘│
│                            ▼                                     │
│ ┌──────────────────────────────────────────────────────────────┐│
│ │ L1: 核心绑定层 (Core Bindings) - Rust 实现                   ││
│ │ __koss_fs_read, __koss_fs_write, __koss_net_fetch,          ││
│ │ __koss_tcp_listen, __koss_hash, __koss_random_bytes 等      ││
│ └──────────────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────────────┘
```

**层级职责说明**：

| 层级 | 名称 | 实现位置 | 职责 | 是否受 Builtin 控制 |
| :--- | :--- | :--- | :--- | :--- |
| L1 | 核心绑定层 | Rust (`src/bindings/`) | 暴露原生 C ABI 函数给 JS | ❌ 仅受 Capability 控制 |
| L2 | 内部系统层 | JS (`src/js_shims/internal/`) | 封装 L1 为易用的同步/异步函数 | ❌ 始终可用（除非沙箱禁止） |
| L3 | 用户态兼容层 | JS (`src/js_shims/`) | 提供 Node/Bun/Deno 风格 API | ✅ 由 Builtin Flags 控制 |

## 2. 核心概念定义

### 2.1 Builtin Flag（特性标志）

控制**哪些兼容层模块对用户可见**的位掩码。与沙箱 Capability 完全解耦。

**命名规范**：`KOSS_BUILTIN_<NAME>`

**存储位置**：`KossInstance` 结构体中的 `builtins` 字段（`uint32_t`）。

### 2.2 内部模块（Internal Module）

以 `koss:internal/` 为前缀的模块，属于 L2 层，**不暴露给最终用户**（用户代码中 `import` 会报错）。仅供 L3 兼容层内部使用。

### 2.3 公开模块（Public Module）

以 `koss:node/`、`koss:bun`、`koss:deno` 为前缀的模块，属于 L3 层，对用户可见，受 Builtin Flags 控制。

### 3.2 模块解析优先级

当用户代码中出现 `import` 或 `require()` 时，模块加载器按以下顺序解析：

1. **检查是否为 `koss:` 协议**：
   - 是 → 进入内置模块解析流程（见第 5 节）
   - 否 → 进入文件系统/`node_modules` 解析流程

2. **内置模块解析流程**：
   - 检查对应的 Builtin Flag 是否启用
   - 未启用 → 抛出 `KossBuiltinError`
   - 已启用 → 返回对应的 JS 模块代码

3. **文件系统解析流程**（非 `koss:` 协议）：
   - 相对路径（`./`、`../`）→ 基于当前文件路径解析
   - 绝对路径（`/` 开头）→ 基于根目录解析
   - 裸模块（`fs`、`express`）→ 在 `node_modules` 中查找

## 4. 特性标志（Builtin Flags）

### 4.1 C ABI 定义

```c
// kossjs.h

typedef enum {
    KOSS_BUILTIN_NONE      = 0,
    KOSS_BUILTIN_NODE      = 1 << 0,  // 启用 koss:node/* 所有模块
    KOSS_BUILTIN_BUN       = 1 << 1,  // 启用 koss:bun
    KOSS_BUILTIN_DENO      = 1 << 2,  // 启用 koss:deno
    KOSS_BUILTIN_KOSS      = 1 << 3,  // 启用 koss:io, koss:crypto, koss:system, koss:data, koss:ffi, koss:worker
    // 预留位 4-31 供未来扩展
    KOSS_BUILTIN_ALL       = 0xFFFFFFFF,
} KossBuiltin;
```

### 4.2 与沙箱 Capability 的交互

```
┌────────────────────────────────────────────────────────────────┐
│                    用户 JS 代码调用                            │
│           例如: Bun.write("/tmp/a.txt", "hello")              │
└────────────────────────┬───────────────────────────────────────┘
                         ▼
┌────────────────────────────────────────────────────────────────┐
│  Step 1: Builtin Flag 检查                                    │
│  └─ KOSS_BUILTIN_BUN 是否启用？                               │
│     ├─ 否 → 抛出 KossBuiltinError:                           │
│     │        "Cannot resolve 'koss:bun' (builtin disabled)"  │
│     └─ 是 → 继续                                              │
└────────────────────────┬───────────────────────────────────────┘
                         ▼
┌────────────────────────────────────────────────────────────────┐
│  Step 2: JS 函数执行 (Bun.write 内部)                        │
│  └─ 调用 koss:internal/fs.writeFileSync(path, data)          │
└────────────────────────┬───────────────────────────────────────┘
                         ▼
┌────────────────────────────────────────────────────────────────┐
│  Step 3: Capability 检查 (Rust 层)                           │
│  └─ FS_WRITE 是否在能力位掩码中？                            │
│     ├─ 否 → 抛出 KossCapabilityError:                       │
│     │        "FS_WRITE capability not granted"               │
│     └─ 是 → 执行实际文件写入                                  │
└────────────────────────────────────────────────────────────────┘
```

### 4.3 组合使用示例

```python
# Python 示例
from kossjs_interface import KossJS

# 场景1: 仅 Bun，无 Node
koss1 = KossJS(
    capabilities=KossJS.KOSS_CAP_ALL_NET | KossJS.KOSS_CAP_ALL_FS,
    builtins=KossJS.KOSS_BUILTIN_BUN,
    stable=True
)
# 可用: import { serve, write } from 'koss:bun'
# 不可用: import { readFileSync } from 'koss:node/fs'

# 场景2: 仅 Deno，无 Node
koss2 = KossJS(
    capabilities=KossJS.KOSS_CAP_SANDBOX,  # 纯计算
    builtins=KossJS.KOSS_BUILTIN_DENO,
    stable=True
)
# 可用: import { crypto } from 'koss:deno'
# 不可用: 任何 IO 操作（沙箱禁止）

# 场景3: Node + Bun，无 Deno
koss3 = KossJS(
    capabilities=KossJS.KOSS_CAP_ALL,
    builtins=KossJS.KOSS_BUILTIN_NODE | KossJS.KOSS_BUILTIN_BUN,
    stable=False
)

# 场景4: 全部禁用（仅基础 JS）
koss4 = KossJS(
    capabilities=KossJS.KOSS_CAP_SANDBOX,
    builtins=KossJS.KOSS_BUILTIN_NONE,
    stable=True
)
# 可用: 纯 JS 计算，无任何内置模块
```

## 5. 模块加载器实现规范

### 5.1 Rust 模块加载器修改

**文件位置**：`src/module_loader.rs`

**新增结构体**：
```rust
pub struct BuiltinModule {
    pub name: &'static str,        // 模块名称，如 "node/fs"
    pub flag: KossBuiltin,          // 所需的 Builtin Flag
    pub source: &'static str,       // JS 源代码
    pub is_internal: bool,          // 是否为内部模块（L2）
}
```

**模块注册表**：
```rust
const BUILTIN_MODULES: &[BuiltinModule] = &[
    // L3: Node 模块
    BuiltinModule { name: "node/fs", flag: KOSS_BUILTIN_NODE, source: include_str!("../js_shims/node_shim/fs.js"), is_internal: false },
    BuiltinModule { name: "node/http", flag: KOSS_BUILTIN_NODE, source: include_str!("../js_shims/node_shim/http.js"), is_internal: false },
    // ... 其他 Node 模块
    
    // L3: Bun 模块
    BuiltinModule { name: "bun", flag: KOSS_BUILTIN_BUN, source: include_str!("../js_shims/bun_shim.js"), is_internal: false },
    
    // L3: Deno 模块
    BuiltinModule { name: "deno", flag: KOSS_BUILTIN_DENO, source: include_str!("../js_shims/deno_shim.js"), is_internal: false },
    
    // L2: 内部模块（无 Flag 要求）
    BuiltinModule { name: "internal/fs", flag: KOSS_BUILTIN_NONE, source: include_str!("../js_shims/internal/fs.js"), is_internal: true },
    BuiltinModule { name: "internal/net", flag: KOSS_BUILTIN_NONE, source: include_str!("../js_shims/internal/net.js"), is_internal: true },
    // ... 其他内部模块
];
```

> **性能备注**：当前使用静态数组 + 线性查找（O(n)），模块数量 ~30 时性能可接受。当模块数量增长到 50+ 时，建议将注册表改为 `HashMap<&str, BuiltinModule>` 以提升查找性能至 O(1)。

**解析逻辑伪代码**：
```rust
fn resolve_builtin(specifier: &str, instance: &KossInstance) -> Result<Module, ModuleError> {
    // 1. 检查是否为 koss: 协议
    if !specifier.starts_with("koss:") {
        return Err(ModuleError::NotBuiltin);
    }
    
    // 2. 提取模块路径（去除 "koss:" 前缀）
    let path = &specifier[5..];  // "node/fs", "bun", "internal/fs"
    
    // 3. 在注册表中查找
    let module = BUILTIN_MODULES.iter().find(|m| m.name == path);
    if module.is_none() {
        return Err(ModuleError::NotFound(format!("Builtin module '{}' not found", specifier)));
    }
    
    // 4. 检查是否为内部模块（L2）—— 用户不可直接导入
    if module.is_internal {
        // 检查调用栈，仅允许来自 L3 Shim 的导入
        if !is_called_from_shim() {
            return Err(ModuleError::Forbidden("Internal modules are not accessible to user code"));
        }
    }
    
    // 5. 检查 Builtin Flag
    if module.flag != KOSS_BUILTIN_NONE {
        let instance_flags = instance.get_builtins();
        if (instance_flags & module.flag) == 0 {
            return Err(ModuleError::BuiltinDisabled(format!(
                "Builtin '{}' requires flag {:?}, current flags: {:?}",
                specifier, module.flag, instance_flags
            )));
        }
    }
    
    // 6. 编译并返回模块
    Ok(compile_module(module.source, specifier))
}
```

### 5.2 调用栈检查（保护 L2 内部模块）

为了确保用户无法直接 `import 'koss:internal/fs'`，需要在 Rust 层检测调用来源：

```rust
fn is_called_from_shim() -> bool {
    // 获取当前 JS 调用栈
    let stack = js::get_call_stack();
    // 检查调用栈中是否包含 "/js_shims/" 路径
    stack.iter().any(|frame| frame.file_name.contains("/js_shims/"))
}
```

> **调试模式**：在开发阶段，可通过编译特性 `KOSS_DEBUG_INTERNAL` 或环境变量 `KOSS_DEBUG_INTERNAL=1` 绕过调用栈检查，允许开发者直接 `require('koss:internal/*')` 进行调试。此开关**仅用于开发**，生产构建中应始终启用调用栈检查。

### 5.2.1 模块加载顺序与循环依赖

KossJS 模块加载器遵循 ES Module 语义：

1. **首次导入时同步执行**：模块在首次 `require()` 时同步解析并执行，结果被缓存。后续 `require()` 同一模块直接返回缓存的导出对象。
2. **循环依赖行为**：循环依赖时，未初始化的导出为 `undefined`（符合 ECMAScript 规范）。CommonJS 风格的 `require()` 同样支持循环依赖，行为与 Node.js 一致（返回已执行部分的导出）。
3. **内置模块无竞态**：所有 `koss:` 内置模块在实例初始化时通过 `include_str!` 预注册为静态字符串，不涉及网络或文件 I/O，解析顺序固定，不存在加载顺序竞态。
4. **模块执行顺序**：模块按依赖图深度优先执行。若模块 A 依赖模块 B，B 会先于 A 执行。若存在循环依赖，先执行到的模块会获得部分导出。

### 5.3 错误类型定义

```rust
pub enum ModuleError {

    NotFound(String),        // 内置模块不存在
    Forbidden(&'static str), // 禁止访问（内部模块）
    BuiltinDisabled(String), // 特性未启用

```

## 6. 内部系统层（L2）实现规范

### 6.1 设计原则

1. **无风格**：不模仿任何运行时的 API 风格（无回调、无 EventEmitter、无 Promise 包装）。
2. **直接暴露**：直接映射 L1 的 `__koss_*` 绑定，仅做必要的类型转换（如将 C 字符串转为 JS String）。
3. **同步优先**：为简化实现，所有 L2 函数均为同步函数（异步由 L3 层包装）。
4. **沙箱透明**：所有 L2 函数调用 L1 时，自动继承当前实例的 Capability 检查。

### 6.2 内部模块列表

| 内部模块 | 导出函数 | 对应 L1 绑定 | 说明 |

## 10. Koss 原生运行时 API（L3）实现规范

### 10.1 设计原则

1. **Koss 自有风格**：不模仿 Node/Bun/Deno，采用 KossJS 独有的简洁 API 设计。
2. **独立控制**：由 `KOSS_BUILTIN_KOSS`（`1 << 3`）独立控制，与 Node/Bun/Deno 互不影响。
3. **零外部依赖**：不依赖 `Buffer`、`TextEncoder`、`TextDecoder`，全部基于纯 `Uint8Array` + 手动字符串操作实现。
4. **同步优先**：所有文件操作均为同步 API，简化使用模型。
5. **大模块设计**：仅 6 个模块，覆盖 I/O、加密、系统、数据、FFI、Worker 全部场景。

### 10.2 模块总览

| 模块 | 导入写法 | 所需 Flag | 说明 |
| :--- | :--- | :--- | :--- |
| `koss:io` | `require('koss:io')` | `KOSS_BUILTIN_KOSS` | 统一 I/O：文件 + 网络 + 流 |
| `koss:crypto` | `require('koss:crypto')` | `KOSS_BUILTIN_KOSS` | 加密与安全：哈希/HMAC/随机/签名 |
| `koss:system` | `require('koss:system')` | `KOSS_BUILTIN_KOSS` | 系统信息：架构/平台/内存/CPU |
| `koss:data` | `require('koss:data')` | `KOSS_BUILTIN_KOSS` | 数据操作：字节编码/Hex/Base64 |
| `koss:ffi` | `require('koss:ffi')` | `KOSS_BUILTIN_KOSS` | 外部函数接口：dlopen/内存管理 |
| `koss:worker` | `require('koss:worker')` | `KOSS_BUILTIN_KOSS` | 工作线程：线程池/消息传递 |

> **捆绑策略说明**：`KOSS_BUILTIN_KOSS` 一次性启用上述全部 6 个模块，不可单独控制。当前版本中，FFI 和 Worker 模块虽然在模块加载层面可见，但在运行时仍受 `stable` 模式和 Capability 的双重约束——即使 `KOSS_BUILTIN_KOSS` 已启用，`stable=true` 时 FFI/Worker 的底层绑定（`_senri_ffi`、`__koss_worker_*`）仍会被剥离，调用时将抛出"not available"错误。若未来需要更细粒度的控制（如单独启用 IO 但禁用 FFI），可拆分为 `KOSS_BUILTIN_KOSS_IO`、`KOSS_BUILTIN_KOSS_FFI` 等子标志。

### 10.7 `koss:ffi` — 外部函数接口模块

**文件位置**：`src/js_shims/koss_shim/ffi.js`

基于 `_senri_ffi` 全局对象，需要 FFI Capability 启用。

> **stable 模式交互**：当 `stable=true` 时，FFI 底层绑定（`_senri_ffi`）会在实例创建阶段被剥离，即使 `KOSS_BUILTIN_KOSS` 已启用。此时 `require('koss:ffi')` 仍会成功（模块可见），但调用 `open()`、`malloc()` 等函数时将抛出 `'FFI not available: _senri_ffi not initialized'` 错误。这是设计上的权衡——模块可见性与运行时能力分离，以保持架构一致性。

| 函数 | 签名 | 返回值 | 说明 |
| :--- | :--- | :--- | :--- |
| `open` | `open(path)` | `NativeLib` | 打开动态库 |
| `dlopen` | `dlopen(path)` | `NativeLib` | 同 `open`（别名） |
| `malloc` | `malloc(size)` | `Pointer` | 分配内存 |
| `free` | `free(pointer)` | `void` | 释放内存 |
| `addressOf` | `addressOf(pointer)` | `number` | 获取指针地址 |
| `createCallback` | `createCallback(fn, signature)` | `Callback` | 创建回调函数 |
| `strerror` | `strerror(errno)` | `string` | 获取错误信息 |

**NativeLib 结构**：
```javascript
{
  fn: function(name, options)     // 获取函数引用
  struct: function(name, options) // 定义结构体
  close: function()               // 关闭动态库
}
```

#### 使用示例

```javascript
var ffi = require('koss:ffi');

// 打开动态库
var lib = ffi.open('user32.dll');  // Windows
// var lib = ffi.open('libm.so');  // Linux

// 获取函数
var MessageBox = lib.fn('MessageBoxA', {
  args: ['pointer', 'string', 'string', 'int'],
  returns: 'int'

// 关闭

```

### 10.8 `koss:worker` — 工作线程模块

**文件位置**：`src/js_shims/koss_shim/worker.js`

基于 `__koss_worker_*` 全局绑定，需要 Worker Capability 启用。

> **数据序列化**：线程间数据传递使用 **JSON 序列化**（`JSON.stringify`/`JSON.parse`），不支持结构化克隆算法。这意味着 `post(data)` 会将数据序列化为 JSON 字符串传输，`receive()` 会将接收到的 JSON 字符串反序列化。不支持传输 `Function`、`Symbol`、`BigInt`、`Map`、`Set`、循环引用等 JSON 不支持的类型。

| 函数 | 签名 | 返回值 | 说明 |
| :--- | :--- | :--- | :--- |
| `createPool` | `createPool(size?)` | `WorkerPool` | 创建线程池（默认 4 线程） |
| `post` | `post(data)` | `void` | 向工作线程发送消息 |
| `receive` | `receive()` | `any \| null` | 接收工作线程消息（非阻塞） |
| `terminate` | `terminate()` | `void` | 终止工作线程 |

**WorkerPool 结构**：
```javascript
{
  execute: function(code) -> Promise,   // 执行 JS 代码
  post: function(data) -> void,         // 发送消息
  receive: function() -> any | null,    // 接收消息（非阻塞）
  terminate: function() -> void,        // 终止线程池
  shutdown: function() -> void          // 关闭线程池
}
```

#### 使用示例

```javascript
var worker = require('koss:worker');

// 创建线程池
var pool = worker.createPool(4);

// 执行代码
pool.execute('var x = 1 + 2; x;').then(function(result) {
  console.log('Result:', result);
});

// 消息传递
pool.post({ task: 'compute', data: [1, 2, 3] });
var msg = pool.receive();

// 清理
pool.shutdown();
```

## 14. 错误处理规范

### 14.1 错误类型层次

```
Error
├── KossError (基类)
│   ├── KossCapabilityError (沙箱权限不足)
│   ├── KossSecurityError (审核回调拒绝)
│   ├── KossBuiltinError (内置模块被禁用)
│   │   ├── BuiltinDisabledError (特性未开启)
│   │   └── InternalModuleError (尝试访问内部模块)
│   ├── KossJSExecutionError (JS 执行错误)
│   └── KossTimeoutError (执行超时)
```

### 14.2 错误消息格式

**BuiltinDisabledError**：
```
BuiltinError: Cannot resolve module 'koss:bun' - Builtin flag KOSS_BUILTIN_BUN is not enabled.
Current builtins: 0x1 (KOSS_BUILTIN_NODE)
To enable: pass builtins=KossJS.KOSS_BUILTIN_BUN when creating instance.
```

**InternalModuleError**：
```
BuiltinError: Cannot import 'koss:internal/fs' - This is an internal module and not accessible to user code.
If you are a developer, ensure the import originates from /js_shims/ directory.
```

### 14.3 各层错误映射

| 错误来源 | 用户看到的错误 | 根本原因 |
| :--- | :--- | :--- |
| L3 用户代码 | `Cannot resolve module 'koss:bun'` | Builtin Flag 未启用 |
| L3 用户代码 | `FS_WRITE capability not granted` | 沙箱 Capability 未授予 |
| L3 内部代码 | `Internal modules not accessible` | 用户直接 `import 'koss:internal/*'` |
| L1 绑定层 | 原生错误消息（透传） | 文件不存在、网络超时等 |

## 附录 C：KossJS 全局对象与架构集成点

### C.1 KossJS 全局对象架构定位

`KossJS` 全局对象作为**运行时锚点**，是所有环境信息的唯一可信来源，与 L1/L2/L3 各层形成完整的集成关系。

```

├─────────────────────────────────────────────────────────────────────────────┤

│  │  globalThis (JS 全局命名空间)                                         │ │
│  │                                                                       │ │

│  │  │   KossJS        │  │  L3 用户态兼容层 (Builtin Flags 控制)      │ │ │
│  │  │   (运行时锚点)   │  │  ┌──────────┐ ┌──────────┐ ┌──────────┐   │ │ │

│  │  └─────────────────────┬───────────────────┘                       │  │ │
```

**KossJS 全局对象的角色**：

| 角色 | 说明 |
| :--- | :--- |
| **运行时锚点** | 所有环境信息（版本、模式）的唯一可信来源 |
| **调试助手** | 用户可在任何 JS 代码中通过 `KossJS` 了解当前运行环境 |
| **兼容层数据源** | L3 模块的 `_compat.kossVersion` 动态引用此对象 |
| **错误消息上下文** | FFI/Worker 等受限模块在报错时引用 `KossJS.version` 和 `KossJS.isStable` |

**安全属性**：

| 属性 | Rust 层 | JS 层加固 | 最终效果 |
| :--- | :--- | :--- | :--- |
| `version` | `READONLY \| ENUMERABLE \| PERMANENT` | `Object.freeze()` | 完全不可变 |
| `runtime` | `READONLY \| ENUMERABLE \| PERMANENT` | `Object.freeze()` | 完全不可变 |
| `isStable` | `READONLY \| ENUMERABLE \| PERMANENT` | `Object.freeze()` | 完全不可变 |
| `KossJS` 本身 | `READONLY \| PERMANENT` | `writable:false, configurable:false` | 不可覆盖、不可删除 |

### C.2 L3 兼容层版本导出规范（`_compat`）

每个 L3 模块（`koss:node/*`、`koss:bun`、`koss:deno`、`koss:*`）**必须**导出一个名为 `_compat` 的标准化对象，包含运行时兼容信息。

**标准导出结构**：

```javascript
// 所有 L3 模块的 _compat 导出必须包含以下字段
const _compat = {
    // 必填字段
    runtime: 'bun' | 'deno' | 'node' | 'koss',  // 兼容的目标运行时
    version: '1.1.42' | '2.0.6' | '20.0.0',     // 兼容的目标运行时版本（硬编码）
    kossVersion: globalThis.KossJS.version,      // 底层 KossJS 内核版本（动态引用）
    target: 'KossJS',                            // 固定值，标识运行环境

    // 可选字段
    isStable: globalThis.KossJS.isStable,        // 稳定模式状态（便于调试）
    build: '20260704',                           // 构建日期（如果有）
};
```

**各模块实现示例**：

| 模块 | `runtime` | `version` | 说明 |
| :--- | :--- | :--- | :--- |
| `koss:bun` | `'bun'` | `'1.1.42'` | 锚定 Bun v1.1.x LTS |
| `koss:deno` | `'deno'` | `'2.0.6'` | 锚定 Deno v2.0.x LTS |
| `koss:node/fs` | `'node'` | `'20.0.0'` | 锚定 Node.js v20.x LTS |
| `koss:io` | `'koss'` | `KossJS.version` | Koss 原生模块，版本与内核一致 |

**用户使用示例**：

```javascript
var ioCompat = require('koss:io')._compat;
console.log('KossJS ' + ioCompat.kossVersion + ' (stable: ' + ioCompat.isStable + ')');

var nodeFsCompat = require('koss:node/fs')._compat;
console.log('Node fs compat: ' + nodeFsCompat.version + ' on KossJS ' + nodeFsCompat.kossVersion);
```

### C.3 FFI/Worker 模块的稳定模式检查

`koss:ffi` 和 `koss:worker` 模块在**模块加载时**检查 `KossJS.isStable`，如果为 `true`，立即抛出明确错误。

**错误消息规范**：

```
[KossJS] FFI module (koss:ffi) is not available in stable mode.
  - Runtime: KossJS v0.1.0-dev.9
  - Current mode: stable (isStable=true)
  - To enable FFI, set 'stable=false' when creating the KossJS instance.
  - Note: FFI requires both KOSS_BUILTIN_KOSS and FFI capability bits.
```

**设计说明**：即使 `KOSS_BUILTIN_KOSS` 启用了 `koss:ffi` 模块，如果 `stable=true`，模块加载本身就会失败。这比"模块可见但调用时报错"更早失败，用户体验更好。
